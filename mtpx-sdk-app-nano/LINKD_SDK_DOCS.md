# @linkd/sdk-typescript Developer Documentation

This document serves as a knowledge base and reference guide for the `@linkd/sdk-typescript` (v0.1.5) framework. It covers the initialization, routing, request context, typed database access, query builder, recommended architectural patterns (Clean Architecture), and testing.

## 1. Initialization

The entry point of the application is the `createService` function. It wires up the HTTP server, the database connection, and links the auto-generated database schema.

```typescript
import { createService } from "@linkd/sdk-typescript";
import type { Schema } from "./db/schema.js";

const service = createService<Schema>({
  name: "users",
  database: { defaultDatabase: "mtpx_sdk_app_nano" },
});

service.afterStart(() => {
  console.log("Service started successfully!");
});

service.start();
```

*Note: The `Schema` interface and table interfaces (e.g., `Users`) are typically auto-generated by the `mtpx db:generate` CLI command.*

## 2. Routing & Request Context (`ctx`)

Endpoints are created using shorthand methods on the `service` instance: `.get()`, `.post()`, `.put()`, `.delete()`. 

Each route handler receives a `ctx` (TypedServiceContext) object:
*   `ctx.body`: The JSON parsed request body. Typed as `unknown` by default.
*   `ctx.query`: A `Record<string, string>` containing URL query parameters. **Note:** All values are strings (e.g., `?active=false` comes as `"false"`).
*   `ctx.params`: A `Record<string, string>` containing path parameters (e.g., `ctx.params.id`).
*   `ctx.db`: The `TypedDatabase<Schema>` instance used to query the database.

### Handling Types in Routes

To avoid `any` and maintain type safety, you can cast the body or parameters directly:

```typescript
import type { Users } from "./db/schema.js";

service.post("/user", async (ctx) => {
  const payload = ctx.body as Partial<Users>;
  return await ctx.db.users.insert(payload);
});
```

When dealing with query parameters, always parse strings to their correct primitive types (e.g., booleans or numbers):
```typescript
const isActive = ctx.query.active === "true";
```

## 3. Database & Query Builder (Fluent API)

The SDK provides a built-in Fluent Query Builder accessible via `ctx.db[tableName]`. 

### A. Insert
If a field has a default value in the database (e.g., `active: true`), you can omit it from the payload. The SDK will cleanly pass only the provided keys.
```typescript
const user = await ctx.db.users.insert({
  id: "uuid-1234",
  name: "John Doe"
});
```

### B. Select & Filtering
Use chaining methods to filter, sort, and paginate. 
*   `.whereEquals(field, value)`
*   `.whereContains(field, value)`
*   `.orderByField(field, direction)`
*   `.limit(number)`
*   `.get()` (Executes the query and returns an array of records)

```typescript
const users = await ctx.db.users
  .whereEquals("active", true)
  .orderByField("name", "asc")
  .limit(10)
  .get();
```

### C. Conditional Queries (`.when`)
The `.when()` method is extremely useful for optional filters (e.g., search queries). It only applies the callback if the first argument evaluates to `true`.
```typescript
const users = await ctx.db.users
  .when(ctx.query.email !== undefined, (q) => q.whereContains("email", ctx.query.email as string))
  .get();
```

### D. Update & Delete
The `.update(data)` method intelligently applies partial updates based *only* on the keys present in the `data` object. 
**Note:** Both `.update()` and `.delete()` resolve to a `Promise<number>`, representing the number of rows affected.

```typescript
// Update
const rowsUpdated = await ctx.db.users
  .whereEquals("id", ctx.params.id as string)
  .update(ctx.body as Partial<Users>);

// Delete
const rowsDeleted = await ctx.db.users
  .whereEquals("id", ctx.params.id as string)
  .delete();
```

## 4. Architectural Best Practices

To keep the routing layer thin and decoupled, it is highly recommended to use **Clean Architecture** patterns, specifically separating database access into **Repositories** and business logic into **Use Cases**.

Because `ctx.db` is tied to the specific request (managing transactions, tenant context, etc.), you should inject `ctx.db` into your repositories on each request.

### Step 1: Define the Repository Interface
```typescript
import type { Users } from "../db/schema.js";

export interface IUserRepository {
  create(data: Partial<Users>): Promise<Users>;
  find(filters: { id?: string; email?: string }): Promise<Users[]>;
  update(id: string, data: Partial<Users>): Promise<number>;
  delete(id: string): Promise<number>;
}
```

### Step 2: Implement the Repository
```typescript
import type { TypedDatabase } from "@linkd/sdk-typescript";
import type { Schema, Users } from "../db/schema.js";
import type { IUserRepository } from "./IUserRepository.js";

export class UserRepository implements IUserRepository {
  constructor(private db: TypedDatabase<Schema>) {}

  async create(data: Partial<Users>): Promise<Users> {
    return await this.db.users.insert(data);
  }
  // ... other methods
}
```

### Step 3: Create Use Cases
```typescript
export class CreateUserUseCase {
  constructor(private userRepository: IUserRepository) {}

  async execute(data: Partial<Users>) {
    if (!data.name) throw new Error("Name is required");
    return await this.userRepository.create(data);
  }
}
```

### Step 4: Wire it together in the Route
```typescript
service.post("/users", async (ctx) => {
  const repository = new UserRepository(ctx.db);
  const useCase = new CreateUserUseCase(repository);
  
  return await useCase.execute(ctx.body as Partial<Users>);
});
```

## 5. Testing (Bun & Clean Architecture)

Thanks to the Clean Architecture approach (separating Use Cases from Repositories), unit testing your business logic becomes trivial. You do not need to mock the `@linkd/sdk-typescript` framework directly. Instead, you create a "Fake" Repository or use mocking functions that implement your domain interfaces.

Since this project runs on **Bun**, you can use the built-in `bun:test` module (which is Jest-compatible).

### Unit Testing a Use Case
Use `mock()` to mimic database calls and assert correct data mapping.

```typescript
import { describe, expect, it, mock } from "bun:test";
import { CreateUserUseCase } from "./CreateUserUseCase.js";
import type { IUserRepository } from "../repositories/IUserRepository.js";
import type { Users } from "../db/schema.js";

describe("CreateUserUseCase", () => {
  it("should successfully create a user", async () => {
    // 1. Mock the repository method
    const mockCreate = mock().mockResolvedValue({
      id: "mock-id",
      name: "Danilo",
    } as Users);

    // 2. Create the Fake Repository
    const fakeRepo: IUserRepository = {
      create: mockCreate,
      find: mock(),
      update: mock(),
      delete: mock(),
    };

    // 3. Inject it into the Use Case
    const useCase = new CreateUserUseCase(fakeRepo);
    const result = await useCase.execute({ name: "Danilo" });

    // 4. Assert
    expect(mockCreate).toHaveBeenCalledTimes(1);
    expect(result.name).toBe("Danilo");
  });
});
```

Execute your tests using the command:
```bash
bun test
```

## 6. OpenAPI / Swagger Support

As of version 0.1.5, the SDK does not natively expose a `/docs` or Swagger endpoint out of the box using `createService`. Auto-generation of OpenAPI documentation would require manual endpoint creation or utilizing external mapping libraries.
